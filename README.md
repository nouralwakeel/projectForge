# projectForge




بناءً على ملف المشروع المرفق (ProjectForge) ومتطلباتك، أرى أن المشروع ممتاز ويمتلك فكرة واضحة المعالم. 

بالنسبة لاختيار التقنيات: **أنصح بشدة باستخدام Laravel كخلفية (Backend) بدلاً من Cloudflare Workers.**
*   **السبب:** نظامك يعتمد بشكل كبير على العلاقات المعقدة في قواعد البيانات (طلاب، مهارات، مشاريع، فرق، تقييمات)، ولارافيل (Laravel) يتفوق بشكل كاسح في إدارة قواعد البيانات العلائقية (Relational Databases) عبر نظام Eloquent ORM، بالإضافة لسهولة بناء واجهات API، والمصادقة (Authentication).
*   *ملاحظة حول الذكاء الاصطناعي (Success Estimator & Matching):* بما أن نطاق العمل (MVP) يسمح بأن تكون الخوارزمية (Rule-based) أو نموذج إحصائي مبسط، يمكن تنفيذ ذلك برمجياً داخل Laravel باستخدام لغة PHP، دون الحاجة لبناء خادم Python منفصل في المرحلة الحالية.

إليك **خطة العمل الشاملة** لإنشاء المشروع من الصفر وحتى التنفيذ النهائي، مصممة لتناسب (Flutter + Laravel):

---

### المرحلة الأولى: التحليل وتصميم قاعدة البيانات (ERD & Architecture)
**المدة المتوقعة: أسبوع واحد**
*   **تصميم قاعدة البيانات (ERD):** بناء الجداول والعلاقات. الجداول الأساسية ستكون:
    *   `Users` (الطلاب/المستخدمين).
    *   `Skills` (المهارات المتاحة لاختيارها).
    *   `User_Skills` (جدول وسيط يربط الطالب بمهاراته مع تحديد مستوى التقييم - Project-DNA).
    *   `Projects` (قائمة المشاريع المقترحة أو المتاحة).
    *   `Project_Tags` (الكلمات المفتاحية للمشاريع لعملية المطابقة).
    *   `Teams` (الفرق المقترحة أو المشكلة).
    *   `Templates` (نماذج الـ JSON الخاصة بـ Sandbox والتي تحتوي على Milestones & Risks).
*   **تحديد الـ API Endpoints:** كتابة توثيق مبدئي (مثل Postman Collection) لجميع المسارات التي سيحتاجها تطبيق Flutter (مثل: تسجيل الدخول، جلب المشاريع، إرسال الاستبيان، جلب خطة Sandbox).

### المرحلة الثانية: تصميم واجهة المستخدم وتجربة المستخدم (UI/UX Design)
**المدة المتوقعة: أسبوع واحد**
*   تصميم الشاشات باستخدام (Figma أو Adobe XD).
*   **الشاشات الأساسية:**
    *   شاشات التسجيل والدخول.
    *   شاشة "استبيان المهارات" لبناء الـ Project-DNA.
    *   لوحة البيانات (Dashboard) لعرض المشاريع الموصى بها.
    *   شاشة تفاصيل المشروع + بيئة المحاكاة (Sandbox) لعرض خطة العمل والمهام.
    *   شاشة "مؤشر احتمالية النجاح" (Success Estimator).
    *   شاشة اقتراح الفريق (Team Matching).

### المرحلة الثالثة: تطوير المنطق البرمجي والخادم (Backend - Laravel)
**المدة المتوقعة: 2 إلى 3 أسابيع**
*   **تأسيس المشروع:** تنصيب Laravel وإعداد قواعد البيانات (MySQL أو PostgreSQL).
*   **نظام المصادقة (Auth):** استخدام `Laravel Sanctum` لعمل API Tokens آمنة لتطبيق Flutter.
*   **بناء الـ APIs الأساسية (CRUD):** برمجة دوال إنشاء الحسابات، وإدخال بيانات الاستبيان.
*   **توليد البيانات الافتراضية (Mock Data):** استخدام `Laravel Seeders & Factories` لتوليد مئات الطلاب الوهميين، والمهارات، والمشاريع، لاختبار خوارزميات المطابقة.
*   **برمجة محرك المطابقة (Recommendation Engine):** كتابة خوارزمية (Rule-based) تقارن بين `User_Skills` و `Project_Tags` وتعطي "درجة تطابق" (Match Score) لترتيب المشاريع لكل طالب.
*   **برمجة مقيّم النجاح (Success Estimator):** كتابة دالة برمجية تحسب احتمالية النجاح بناءً على معادلة رياضية (مثلاً: إذا كانت مهارات الطالب + مهارات الفريق تغطي 80% من متطلبات المشروع، وتصنيف المشروع "صعب"، فإن نسبة النجاح 75%).
*   **برمجة الـ Sandbox:** تجهيز نظام يُرجع بيانات (Milestones & Risks) مهيكلة بصيغة JSON بناءً على تصنيف المشروع (تطبيق ويب، ذكاء اصطناعي، شبكات.. إلخ).

### المرحلة الرابعة: تطوير واجهات التطبيق (Frontend - Flutter)
**المدة المتوقعة: 3 أسابيع (تتداخل مع المرحلة الثالثة)**
*   **تأسيس تطبيق Flutter:** بناء الهيكلية الأساسية واستخدام إدارة حالة (State Management) مناسبة مثل `GetX` أو `Bloc` أو `Riverpod`.
*   **برمجة واجهات المستخدم:** تحويل تصميم Figma إلى أكواد Flutter تفاعلية.
*   **الربط مع الخادم (API Integration):** 
    *   ربط شاشات التسجيل والاستبيان.
    *   استقبال بيانات المشاريع الموصى بها وعرضها بشكل بطاقات جذابة.
    *   استقبال بيانات الـ Sandbox (JSON) وعرضها كـ "شجرة مهام" (Timeline أو Gantt Chart مبسط).
    *   عرض نسبة "احتمالية النجاح" باستخدام رسومات بيانية (Circular Progress Indicator أو Gauges).

### المرحلة الخامسة: الدمج والاختبار (Integration & Testing)
**المدة المتوقعة: أسبوع واحد**
*   **اختبار الخوارزميات (Logic Testing):** التأكد من أن التوصيات منطقية وأن الطالب الذي يمتلك مهارات (Flutter, Dart) تظهر له مشاريع تطبيقات الموبايل وليس مشاريع تحليل البيانات.
*   **اختبار الأداء والتجاوب (UI Testing):** التأكد من عمل تطبيق Flutter بشكل ممتاز على أجهزة Android و iOS وبمقاسات شاشات مختلفة.
*   **اكتشاف الأخطاء (Bug Fixing):** معالجة أي أخطاء تظهر عند ربط Flutter مع Laravel.

### المرحلة السادسة: التنفيذ النهائي والنشر (Deployment & Launch)
**المدة المتوقعة: أسبوع واحد**
*   **استضافة الـ Backend:** 
    *   رفع مشروع Laravel على خادم مثل DigitalOcean, AWS، أو استضافة سحابية عادية تدعم PHP/MySQL.
    *   إعداد شهادة الأمان (SSL) لضمان اتصال مشفر (HTTPS) بين التطبيق والخادم.
*   **نشر تطبيق Flutter:**
    *   استخراج النسخة النهائية (APK/AAB) للأندرويد، و (IPA) للآيفون.
    *   رفع التطبيق على Google Play Store و Apple App Store (إذا كان المشروع يستهدف النشر التجاري)، أو تجهيزه كنسخة تجريبية (Demo) لعرض التخرج.
*   **العرض التوضيحي (Demo Preparation):** تجهيز سيناريو العرض؛ إنشاء حساب طالب جديد، تعبئة مهارات محددة، استعراض كيف يقترح النظام مشروعاً دقيقاً، وكيف يولد خطة Sandbox، وكيف يقرأ نسبة النجاح.

---

### لماذا هذه الخطة هي الأنسب لـ MVP (نطاق العمل الواقعي)؟
1.  **السرعة والفعالية:** استخدام Laravel سيختصر عليك أسابيع من كتابة أكواد إدارة قواعد البيانات والمصادقة مقارنة ببيئات أخرى، مما يتيح لك التركيز على "الخوارزميات" و "التطبيق".
2.  **التوافق مع متطلباتك:** الخطة تغطي تماماً ما جاء في ملف PDF (استبيان، Sandbox، Recommendation، Success Estimator).
3.  **قابلية التوسع مستقبلاً:** إذا أردت لاحقاً إضافة ذكاء اصطناعي حقيقي (Machine Learning / Logistic Regression) بدلاً من الـ Rule-based، يمكنك ببساطة بناء (Microservice) صغيرة بـ Python وربطها مع Laravel عبر API، دون الحاجة لهدم النظام الأساسي.

بناءً على طلبك، قمت بتصميم **مخطط قواعد بيانات (ERD - Entity Relationship Diagram)** موسع وشامل وفق المعايير الأكاديمية (الاعتماد على التطبيع Normalization حتى المستوى الثالث 3NF، واستخدام الجداول الوسيطة Pivot Tables لتجنب التكرار).

لقد قمت بتضمين جميع التعديلات التي طلبتها في جدول المستخدمين، وقمت بفصل "التخصص" في جدول مستقل لضمان سلامة البيانات (Data Integrity).

إليك هيكلية الجداول الأكاديمية لمشروع **ProjectForge**:

---

### 1. الإدارة والمستخدمين (User Management)

**جدول التخصصات الأكاديمية (`majors`)**
*(تم فصله عن جدول المستخدمين لتوحيد أسماء التخصصات وتجنب الأخطاء الإملائية)*
*   `id` (Primary Key - Integer)
*   `name` (String) - اسم التخصص (مثل: هندسة برمجيات، علوم حاسب).
*   `code` (String) - رمز التخصص (مثل: SE, CS).

**جدول المستخدمين (`users`)** *(تم تطبيق كافة شروطك هنا)*
*   `id` (Primary Key - BigInt)
*   `student_id` (String - Unique) - **الرقم الشخصي / الجامعي.**
*   `first_name` (String) - **الاسم الأول.**
*   `last_name` (String) - **الاسم الأخير.**
*   `email` (String - Unique) - البريد الإلكتروني.
*   `password` (String) - كلمة المرور (مشفرة).
*   `gender` (Enum: 'male', 'female') - **الجنس.**
*   `date_of_birth` (Date) - **تاريخ الميلاد.**
*   `major_id` (Foreign Key -> `majors.id`) - **التخصص.**
*   `academic_level` (Integer أو Enum) - **المستوى الأكاديمي** (مثل: 4 للرابع، أو 'Senior').
*   `role` (Enum: 'student', 'advisor', 'admin') - دور المستخدم (طالب، مشرف، مدير).
*   `created_at` / `updated_at` (Timestamps).

---

### 2. محرك المهارات والتوصية (Skills & Project-DNA)

**جدول المهارات المرجعية (`skills`)**
*(يحتوي على جميع المهارات المتاحة في النظام)*
*   `id` (Primary Key - Integer)
*   `name` (String) - اسم المهارة (Flutter, Python, Machine Learning).
*   `category` (String) - التصنيف (Frontend, Backend, AI, Soft Skills).

**جدول ملف مهارات الطالب (`user_skills`) - (يُمثل الـ Project-DNA)**
*(جدول وسيط Many-to-Many يربط الطالب بالمهارات التي يمتلكها بناءً على الاستبيان)*
*   `id` (Primary Key - BigInt)
*   `user_id` (Foreign Key -> `users.id`)
*   `skill_id` (Foreign Key -> `skills.id`)
*   `proficiency_level` (Integer: 1 to 5) - مستوى التقييم/الإجادة في هذه المهارة.

---

### 3. المشاريع والمطابقة (Projects & Matching)

**جدول المشاريع (`projects`)**
*   `id` (Primary Key - BigInt)
*   `title` (String) - عنوان المشروع.
*   `description` (Text) - وصف المشروع.
*   `type` (String) - نوع المشروع (تطبيق موبايل، موقع ويب، نظام ذكاء اصطناعي).
*   `difficulty_level` (Integer: 1 to 5) - مستوى الصعوبة (يفيد في خوارزمية التقييم Success Estimator).
*   `advisor_id` (Foreign Key -> `users.id` - Nullable) - المشرف الأكاديمي المقترح.
*   `status` (Enum: 'available', 'taken', 'completed') - حالة المشروع.

**جدول متطلبات المشروع من المهارات (`project_skills`)**
*(جدول وسيط يربط المشروع بالمهارات المطلوبة لإنجازه)*
*   `id` (Primary Key - BigInt)
*   `project_id` (Foreign Key -> `projects.id`)
*   `skill_id` (Foreign Key -> `skills.id`)
*   `weight` (Decimal) - **الوزن الأهمية:** (مثلاً مهارة Flutter وزنها 80%، ومهارة Firebase وزنها 20% لتحديد مدى التطابق).

---

### 4. فرق العمل (Teams)

**جدول الفرق (`teams`)**
*   `id` (Primary Key - BigInt)
*   `name` (String) - اسم الفريق.
*   `project_id` (Foreign Key -> `projects.id`) - المشروع الذي اختاره الفريق.
*   `is_approved` (Boolean) - حالة الموافقة على الفريق.

**جدول أعضاء الفريق (`team_members`)**
*(جدول وسيط يربط الطلاب بالفرق)*
*   `id` (Primary Key - BigInt)
*   `team_id` (Foreign Key -> `teams.id`)
*   `user_id` (Foreign Key -> `users.id`)
*   `role_in_team` (String) - دور الطالب (Team Leader, Developer, UI/UX).

---

### 5. بيئة المحاكاة وخطط التنفيذ (Sandbox: Milestones & Risks)
*ملاحظة أكاديمية: رغم أن المستند نص على إرجاع الخطة بصيغة JSON، إلا أنه في قواعد البيانات الأكاديمية يجب تخزين البيانات بشكل مهيكل (الجداول التالية) ثم يقوم الباك-إند (Laravel) بتحويلها إلى JSON عند إرسالها للواجهات.*

**جدول المراحل التنفيذية (`milestones`)**
*   `id` (Primary Key - BigInt)
*   `project_id` (Foreign Key -> `projects.id`)
*   `title` (String) - اسم المرحلة (مثل: بناء واجهات المستخدم).
*   `description` (Text) - تفاصيل المرحلة.
*   `estimated_days` (Integer) - المدة المتوقعة بالأيام.
*   `order_sequence` (Integer) - الترتيب الزمني للمرحلة (1، 2، 3...).

**جدول المخاطر المحتملة (`risks`)**
*   `id` (Primary Key - BigInt)
*   `project_id` (Foreign Key -> `projects.id`)
*   `risk_description` (Text) - وصف الخطر (مثل: عدم توافق إصدارات المكتبات).
*   `impact_level` (Enum: 'Low', 'Medium', 'High') - مستوى التأثير.
*   `mitigation_plan` (Text) - خطة تدارك الخطر.

---

### 6. مقيّم احتمالية النجاح (Success Estimator)

**جدول التقييمات التقديرية (`success_estimations`)**
*(لتخزين نتائج الخوارزمية التي تحسب فرصة النجاح بناءً على المهارات وصعوبة المشروع)*
*   `id` (Primary Key - BigInt)
*   `team_id` (Foreign Key -> `teams.id` - Nullable) - في حال كان التقييم لفريق.
*   `user_id` (Foreign Key -> `users.id` - Nullable) - في حال كان التقييم لطالب فردي مع مشروع.
*   `project_id` (Foreign Key -> `projects.id`)
*   `success_probability` (Decimal) - النسبة المئوية لفرصة النجاح (مثلاً 85.5%).
*   `calculated_at` (Timestamp) - وقت إجراء التقييم.
*   `factors_log` (JSON) - حقل لتخزين سبب هذه النسبة (مثلاً: توافق المهارات 90% + مستوى الصعوبة 4).

---

### ملخص العلاقات (Relationships Summary):
1.  **المستخدم والتخصص:** علاقة (One-to-Many). التخصص الواحد يضم عدة طلاب.
2.  **المستخدم والمهارة:** علاقة (Many-to-Many) عبر جدول `user_skills` (وهو الـ Project-DNA).
3.  **المشروع والمهارة:** علاقة (Many-to-Many) عبر جدول `project_skills` (تُستخدم للمطابقة Matching).
4.  **المشروع والمراحل (Milestones):** علاقة (One-to-Many).
5.  **المشروع والمخاطر (Risks):** علاقة (One-to-Many).
6.  **الفريق والمستخدم:** علاقة (Many-to-Many) عبر جدول `team_members`.

**هل هذا المخطط يلبي رؤيتك الأكاديمية للمشروع؟ إذا كان مناسباً، يمكننا الانتقال إلى الخطوة التالية وهي تحديد دوال الـ API (Endpoints) المطلوبة للربط بين Laravel و Flutter.**
